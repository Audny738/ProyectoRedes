¿Cómo podemos mantener nuestras aplicaciones seguras?

Como ya vimos anteriormente, existen innumerables amenazas hacia la web y, por lo tanto, debemos asegurar nuestro sitio con las medidas necesarias. A continuacion se enlistan ciertas medidas con base al enfoque que se quiere proteger.

Asegurar el Contenido

  • Política de Seguridad de Contenido

                <meta
                http-equiv="Content-Security-Policy"
                content="default-src 'self'; img-src https://*; child-src 'none';" />

Es una capa extra de seguridad que ayuda a detectar y mitigar cietos tipos de ataque como el Cross-Site Scripting (XSS) o la inyección de datos. La forma de integrarlo se muestra en el snippet adjunto.

Asegurar la Conexión

  • Seguridad en la capa de Transporte (TLS)

Es un estánar que permite a dos aplicaciones o dispositivos conectados a la red el intercambio de información privada. Un caso de uso común de TLS es encriptar la comunicación entre aplicaciones web y servidores.

Para que una aplicación web use el TLS debe tener un certificado TLS, también conocido con certificado SSL, instalado en tu servidor origen. Una autoridad certificada es la encargada de verificar a la persona. Este certificado contiene información importante como: quién el el dueño del dominio y la llave pública del servidor. Como se ha mencionado a SSL, vale la pena hablar un poco más a fondo del tema.

Secure Socket Layer (SSL)

Este era un viejo protocolo de seguridad para crear un enlace de red encriptado entre el servidor y el cliente, asegurando que toda la información pase de manera rápida y segura. SSL es el predecesor de TLS y un sitio web que tiene HTTPS en su URL implementa SSL/TLS.

Al igual que TLS, existen certificados SSL. Estos certificados son como una tarjeta de identificación que valida que eres quien dices ser.

  • HTTPS

El HyperText Transfer Protocol Secure (HTTPS) es una versión encriptada del protocolo HTTP. Utiliza SSL o TLS para encriptar toda la comunicación entre un cliente y un servidor. Esto conexión segura permite a los clientes estar seguros que están conectados al servidor correcto y están seguros de intercambiar información sensible. En el siguiente video se puede encontrar una explicación más profunda.

  • HTTP Strict-Transport-Security

Es un encabezado de HTTP que deja a un sitio web especificar que solo puede ser accedida utilizando HTTPS. 


              Strict-Transport-Security: max-age=<expire-time>
              Strict-Transport-Security: max-age=<expire-time>; includeSubDomains
              Strict-Transport-Security: max-age=<expire-time>; includeSubDomains; preload

  • Transparecia de Certificados

Es un framework diseñado para proteger y monitorear emisiones erróneas de certificados. Los certificados recién emitidos se "registran" para ejecutar públicamente, a menudo son registros CT independientes que contienen un registro encriptado que asegura los certificados TLS emitidos.

  • Contexto Seguro

Un contexto seguro es una ventana en la cual hay una confianza razonable de que el contenido ha sido entregado seguro y limita a los que el potencial para la comunicación con contextos no es seguro. Un ejemplo de ello son las APIs Web que solo son accesibles bajo un contexto seguro.

Seguridad de Datos

  • HTTP Cookies

Pequeña pieza de información que el servidor le envía al navegador web del usuario. El navegador puede almacenarlo y enviarlo de vuelta con otras peticiones al servidor. Normalmente, es usuado para ver si dos solicitudes vienen del mismo navegador.

Integridad

  • Política del mismo origen

Es un mecanismo de seguridad crítico que restringe como un documento o scrpt cargado desde el origen puede interactuar con un recurso de otro origen aislando los potenciales documentos maliciosos.

Protección de los clickjack

Clickjacking es un ataque que se hospeda en la interfaz del sitio, manipulando a los usuarios a apretarlo para dirigirlos a enlaces maliciosos. Aquí los atacantes incrustan sus enlaces maliciosos en botones o páginas legítimas del sitio web y de esta forma obtiene la información confidencial del usuario.

  • HTTP X-Frame-Options

Es utilizado para indicar si el navegador puede renderizar una página en un <frame>, <iframe>, <embed> u <object>. Los sitios web pueden evitar ataques asegurando que su contenido no está embebido dentro de otros sitios.

  • CSP: frame-ancestors

La directiva CSP frame-ancestors especifica padres válidos que pueden insertar una página usuando <frame>, <iframe>, <embed>, <object> o <applet>.
